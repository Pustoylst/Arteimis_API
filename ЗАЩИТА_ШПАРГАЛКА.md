# Шпаргалка для защиты Arteimis_API

## Суть проекта в одном предложении
REST API для управления проектами, задачами и комментариями в команде разработки. Как трекер типа Jira или Trello.

---

## Основные технологии

- **Python 3.13** — язык программирования
- **Django 6.0.4** — веб-фреймворк
- **Django REST Framework** — библиотека для создания REST API
- **django-filter 25.2** — фильтрация запросов
- **drf-spectacular** — автодокументация (Swagger, ReDoc)
- **SQLite** — база данных (локальная разработка)

---

## Три ключевые модели (БД)

### Project (Проект)
- **creator** — владелец, который создал проект
- **participants** — список участников (ManyToMany)
- **created_at** — когда был создан
- При создании creator автоматически становится участником

### Task (Задача)
- **project** — к какому проекту относится
- **author** — кто создал задачу (может менять только описание)
- **assignee** — кто исполняет (может менять статус и приоритет)
- **priority** — LOW, MEDIUM, HIGH, CRITICAL
- **status** — TODO, IN_PROGRESS, REVIEW, DONE, BLOCKED
- **deadline** — дедлайн задачи

### TaskComment (Комментарий)
- **task** — к какой задаче относится
- **author** — кто написал
- **text** — текст комментария
- Видны только участникам проекта

---

## Права доступа (самая важная часть!)

### На проекты:
- **Владелец**: может редактировать проект, управлять участниками, редактировать/удалять все задачи
- **Участник**: может видеть проект и его задачи, создавать новые задачи

### На задачи:
- **Владелец проекта**: может всё (менять любые поля, удалять)
- **Исполнитель задачи**: может менять только **status** и **priority**
- **Автор задачи** (если не владелец): может менять только **description**
- **Другие**: нечего не могут менять

### На комментарии:
- **Участник проекта**: может читать
- **Автор** или **владелец проекта**: могут редактировать/удалять

## Основные эндпоинты

```
POST   /api/register/                           — регистрация пользователя
GET    /api/projects/                           — список проектов (фильтрованный)
POST   /api/projects/                           — создание проекта
POST   /api/projects/{id}/add_participant/      — добавить участника
POST   /api/projects/{id}/remove_participant/   — удалить участника

GET    /api/tasks/                              — список задач (с фильтрацией)
POST   /api/tasks/                              — создание задачи
PATCH  /api/tasks/{id}/                         — частичное изменение

GET    /api/comments/                           — список комментариев
POST   /api/comments/                           — добавить комментарий

/api/schema/                                    — OpenAPI схема (JSON)
/api/docs/swagger/                              — интерактивная документация
/api/docs/redoc/                                — альтернативная документация
```

## Как работает DRF (Django REST Framework)

1. **Serializer** — преобразует Django-модель в JSON и обратно
   - Может валидировать данные (проверять бизнес-правила)
   - В `TaskSerializer.validate()` проверяем, что исполнитель — участник проекта

2. **ViewSet** — обработчик API эндпоинтов (это как контроллер)
   - `ModelViewSet` даёт все CRUD методы автоматически
   - `get_queryset()` фильтрует данные (текущий юзер видит только свои)
   - `perform_create()` срабатывает при POST — устанавливает автора

3. **Router** — автоматически создаёт URL маршруты из ViewSet-ов
   - `register("projects", ProjectViewSet)` → `/api/projects/`, `/api/projects/{id}/`, и т.д.

4. **Permissions** — правила доступа к объектам
   - `IsAuthenticated` — требует аутентификации
   - Кастомные классы типа `IsProjectCreator`, `TaskObjectPermission` проверяют права

## Типовые вопросы и ответы

**Q: Почему именно DRF?**
A: DRF предоставляет готовые компоненты для быстрого создания REST API (Serializers, ViewSets, Permissions, Pagination, Authentication), что экономит время и уменьшает количество кода.

**Q: Как реализованы права доступа?**
A: Через object-level permissions в DRF. Класс `TaskObjectPermission` проверяет каждый метод HTTP запроса и определяет, кто может делать что с задачей.

**Q: Зачем проверять, что исполнитель — участник проекта?**
A: Чтобы нельзя было назначить задачу человеку, который даже не знает про этот проект. Это бизнес-правило для корректности данных.

**Q: Почему creator проекта автоматически становится участником?**
A: Потому что владелец проекта должен видеть все задачи. Это реализуется в методе `Project.save()`.

**Q: Зачем нужна фильтрация?**
A: Чтобы пользователи могли быстро найти задачи по проекту, статусу, приоритету или дедлайну без загрузки всех данных.
